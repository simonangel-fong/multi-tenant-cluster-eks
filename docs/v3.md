# Version 3

## Goal

Reframe the project from "EKS cluster with tools installed" to a **production-style application platform** with clear ownership between platform and workload teams.

## Thesis

> An EKS-based application platform demonstrating clear ownership separation between platform and workload teams — providing compute, ingress, secrets, TLS, DNS, GitOps, progressive delivery, and observability, with two demo workloads proving the platform is a product, not a bespoke setup.

## Capabilities offered

| Layer         | Capability                         | Tools                                                                  |
| ------------- | ---------------------------------- | ---------------------------------------------------------------------- |
| Compute       | Node autoscaling, workload classes | Karpenter (general + database NodePools)                               |
| Storage       | Block storage with IOPS tiers      | EBS CSI, `gp3` + `gp3-iops` StorageClasses                             |
| Network       | Service mesh, ingress, DNS, TLS    | Istio ambient, Gateway API, external-dns, cert-manager                 |
| Security      | Secrets, policy, network isolation | ESO (Pod Identity), Kyverno, NetworkPolicy + Istio AuthorizationPolicy |
| Delivery      | GitOps, progressive delivery       | ArgoCD, Argo Rollouts                                                  |
| Observability | Metrics, logs, dashboards, alerts  | Prometheus, Loki, Grafana, Alertmanager                                |
| Tenancy       | Team onboarding primitive          | TeamNamespace pattern (Helm + ArgoCD)                                  |

## Ownership model

- **Platform team** owns the capabilities above, the tenancy abstraction, and the contracts.
- **Workload teams** own their apps, dashboards, alerts, SLOs, and on-call for their services.
- **Identity source**: GitHub teams → OIDC group claims → ArgoCD + Grafana RBAC. Documented path to IAM Identity Center for larger orgs.

## Cross-cutting design decisions

- **Notifications**: one Slack app, label-based routing (`team=X`) across ArgoCD, Alertmanager, Rollouts. Same shape everywhere.
- **Policy**: Kyverno starter set, audit → enforce rollout.
- **Multi-environment**: single cluster, `-dev` + `-prod` namespaces. Cluster-per-environment covered in a separate project.
- **Tenancy**: teams onboard via ~3 YAML files under `argocd/tenants/`. Namespace, quota, RBAC, NetworkPolicy, Grafana folder, alert routing all generated.
- **Depth**: Tier 2 across the board (complete but not exhaustive). Deeper items go on the roadmap.

## Documentation deliverables

Three concise, one-page docs:

- `docs/capabilities.md` — capability catalog with owner + consumption pattern
- `docs/workload-contract.md` — what teams must ship to consume the platform
- `docs/onboarding.md` — 10-step "onboard a team" runbook with time budget

## Roadmap (deferred, documented in README)

- Cost visibility (OpenCost)
- Backup / DR (Velero)
- SLO framework (Sloth / Pyrra)
- Distributed tracing (Tempo)
- Long-term metrics (Thanos / Mimir)
- IAM Identity Center for AWS SSO
- Cluster-per-environment (covered in separate project)

## Success criteria

- Reviewer opens repo and can name what the platform offers within 60 seconds
- Onboarding runbook takes a new team from zero to first HTTPRoute in one afternoon
- Second app onboards with zero platform code changes
- Every capability has a stated owner and consumption pattern
- Every deferred capability is named in the roadmap, not silently missing
