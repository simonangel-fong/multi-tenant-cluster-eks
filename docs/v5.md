## Goal

Refactor the project into a **multi-tenant cluster** with a clear boundary between the **platform** and its **tenants**.

Three guiding questions frame every phase:

1. What capabilities does the platform offer?
2. How do tenants consume those capabilities?
3. For each capability, how do platform and tenant interact at the boundary?
   - Example: when a tenant needs additional GPU nodes, how does the platform extend compute capacity?
   - Example: when a tenant needs access to S3, how does the platform enable it?

## Phases

### Phase 00 — Review baseline capabilities

**Purpose.** Establish the ground truth for what the cluster provides today and decide, for each capability, whether it belongs in the minimum viable multi-tenant platform (**must**) or is a value-add that can be deferred (**optional**). Everything downstream — responsibility model, onboarding contract, tenant demos — is built on this table.

**Classification rules.**

- **Must** — the platform cannot deliver its promised isolation, availability, or operability without it. Removing it breaks the tenancy model.
- **Optional** — improves developer experience, efficiency, or governance, but the platform still functions if it is deferred. Optional capabilities may still ship on day one; the label is about _necessity_, not _presence_.

**Capability inventory.**

| Layer         | Capability                          | Tooling                                       | Class    | Rationale                                                                                         |
| ------------- | ----------------------------------- | --------------------------------------------- | -------- | ------------------------------------------------------------------------------------------------- |
| Compute       | Node autoscaling & node classes     | Karpenter (`platform`, `general`, `database`) | Must     | Tenants need a stable placement contract; static node groups do not scale to multi-tenant demand. |
| Compute       | GPU node class                      | Karpenter (`gpu`)                             | Optional | Only required once an ML tenant onboards. Design for it now, provision on demand.                 |
| Storage       | Block storage — general             | EBS CSI, `gp3` StorageClass                   | Must     | Default PVC path for stateless and stateful workloads.                                            |
| Storage       | Block storage — high IOPS           | EBS CSI, `gp3-iops` StorageClass              | Must     | Database tenants require predictable IOPS and `Retain` reclaim policy.                            |
| Storage       | Object storage access (S3)          | IRSA / Pod Identity + tenant-owned buckets    | Must     | Common tenant requirement (logs, artifacts, datasets); the boundary must be defined up front.     |
| Storage       | Shared filesystem                   | EFS CSI                                       | Optional | No current tenant requires RWX. Defer until a use case emerges.                                   |
| Network       | Pod networking + NetworkPolicy      | VPC CNI (NP enabled)                          | Must     | Foundation of tenant network isolation.                                                           |
| Network       | Service mesh & mTLS                 | Istio ambient                                 | Must     | Cross-tenant traffic must be authenticated and encrypted by default.                              |
| Network       | Ingress (Gateway API)               | Istio ambient + AWS Load Balancer Controller  | Must     | Single, opinionated ingress path that tenants attach to via `HTTPRoute`.                          |
| Network       | DNS automation                      | external-dns                                  | Must     | Tenant hostnames must resolve without manual Route 53 edits.                                      |
| Network       | TLS certificates                    | cert-manager                                  | Must     | Automatic certificate issuance for tenant hostnames.                                              |
| Security      | Secrets management                  | External Secrets Operator (ESO)               | Must     | Removes the need for tenants to embed AWS credentials or manage `Secret` lifecycle by hand.       |
| Security      | Admission policy                    | Kyverno                                       | Must     | Enforces the tenancy contract (labels, resource limits, image sources) at admission.              |
| Security      | Network isolation                   | NetworkPolicy + Istio AuthorizationPolicy     | Must     | Default-deny between tenant namespaces is a non-negotiable baseline.                              |
| Security      | Image signing / provenance          | Cosign + Kyverno `verify-images`              | Optional | Meaningful only once a signing pipeline exists. Defer until supply chain is in scope.             |
| Observability | Metrics                             | Prometheus + Grafana Alloy                    | Must     | Platform needs cluster-level metrics; tenants need per-workload metrics.                          |
| Observability | Logs                                | Loki (S3 backend) + Alloy                     | Must     | Tenant log isolation is part of the tenancy contract.                                             |
| Observability | Dashboards                          | Grafana                                       | Must     | Delivery surface for both platform and tenant observability.                                      |
| Observability | Distributed tracing                 | Tempo (or OTLP collector)                     | Optional | Ambient mesh emits trace-ready data, but no tenant currently consumes traces.                     |
| Observability | Long-term metrics                   | Thanos / Mimir                                | Optional | 7-day Prometheus retention is sufficient for the current scope.                                   |
| Delivery      | GitOps (platform)                   | ArgoCD                                        | Must     | Platform components are managed as code; ArgoCD is the reconciliation loop.                       |
| Delivery      | GitOps (tenant apps)                | ArgoCD `Application` per tenant               | Must     | Onboarding a tenant means registering an `Application`; no imperative kubectl path.               |
| Delivery      | Progressive delivery                | Argo Rollouts                                 | Optional | Offered as a shared capability; tenants opt in. Platform itself does not require it.              |
| Tenancy       | Namespace + quota + default-deny NP | Helm chart / ArgoCD `ApplicationSet`          | Must     | The onboarding primitive. Every capability above assumes this scaffold exists.                    |

**Outcome.**

The **Must** rows define the scope of Phases 01–05. The **Optional** rows are captured here so their absence reads as _deferred_, not _forgotten_ — they resurface in the roadmap section of the final onboarding manual (Phase 07).

### Phase 02 — Storage capabilities

**Purpose.** Define the storage contract between the platform and tenants for two distinct concerns: **block storage** (per-pod PVCs backed by EBS) and **object storage** (tenant-owned S3 buckets accessed by pods). The two share nothing operationally — different providers, different consumption patterns, different failure modes — so they are modeled separately.

#### Responsibility model

| Concern                                                | Platform | Tenant | Notes                                                                                                                                |
| ------------------------------------------------------ | :------: | :----: | ------------------------------------------------------------------------------------------------------------------------------------ |
| CSI driver lifecycle                                   |    ✅    |        | `aws-ebs-csi-driver` installed as an EKS-managed add-on with Pod Identity.                                                           |
| StorageClass definitions                               |    ✅    |        | `gp3` (default) and `gp3-iops`, GitOps-managed under [argocd/platform-capabilities/storage/](argocd/platform-capabilities/storage/). |
| PVC creation, sizing, mounting                         |          |   ✅   | Tenants declare PVCs with a `storageClassName` and access mode.                                                                      |
| Volume expansion                                       |          |   ✅   | Enabled at the StorageClass level; tenants patch PVC `spec.resources` themselves.                                                    |
| S3 bucket provisioning                                 |    ✅    |        | Buckets are Terraform-managed alongside the cluster (see Loki bucket precedent in [infra/16-eks-loki.tf](infra/16-eks-loki.tf)).     |
| S3 bucket contents (objects, prefixes)                 |          |   ✅   | Tenants own everything inside the bucket.                                                                                            |
| Pod → S3 identity (IAM role, Pod Identity association) |    ✅    |        | Platform creates the role and binds it to the tenant's ServiceAccount.                                                               |
| ServiceAccount → pod wiring                            |          |   ✅   | Tenants set `spec.serviceAccountName` on their workload.                                                                             |

#### Block storage — StorageClass matrix

| Class             | Provisioner       | Type | IOPS / Throughput       | Reclaim  | Binding                | Expansion | Use                                                                                               |
| ----------------- | ----------------- | ---- | ----------------------- | -------- | ---------------------- | --------- | ------------------------------------------------------------------------------------------------- |
| `gp3` _(default)_ | `ebs.csi.aws.com` | gp3  | AWS defaults            | `Delete` | `WaitForFirstConsumer` | ✅        | Default PVC path for stateless caches, scratch space, general-purpose stateful workloads.         |
| `gp3-iops`        | `ebs.csi.aws.com` | gp3  | 10 000 IOPS / 500 MiB/s | `Retain` | `WaitForFirstConsumer` | ✅        | Databases, WALs, write-heavy stateful workloads. `Retain` protects data on accidental PVC delete. |

Design principles:

- **Storage class is a tenant choice, not a workload class inheritance.** A pod on the `database` node class does not automatically get `gp3-iops`; the tenant declares it. Coupling the two would prevent legitimate combinations (e.g., a database that only needs `gp3`, or a stateless workload on `general` that wants provisioned IOPS).
- **`WaitForFirstConsumer` on both classes.** The volume is provisioned in the same AZ as the pod that will mount it, avoiding cross-AZ scheduling failures.
- **`Retain` on `gp3-iops` is a safety net, not a backup strategy.** Orphaned volumes must be cleaned up manually. Backup (Velero, EBS snapshots) is a Phase 05 / roadmap item.
- **No RWX today.** EFS is called out as optional in Phase 00 and will be added only when a tenant requires shared filesystem access.

#### Object storage (S3) — access pattern

S3 access is granted via **EKS Pod Identity**, the same mechanism already in production for the Loki bucket ([infra/16-eks-loki.tf](infra/16-eks-loki.tf)). The pattern is:

1. Platform provisions the bucket in Terraform (SSE enabled, public access blocked, lifecycle rules if applicable).
2. Platform creates an IAM role scoped to that bucket ARN with least-privilege actions.
3. Platform creates a `PodIdentityAssociation` binding the role to a `(namespace, serviceAccountName)` pair the tenant will use.
4. Tenant creates the `ServiceAccount` in their namespace and references it in their pod spec.

No static AWS credentials are ever mounted into a tenant pod. Credentials are vended by the Pod Identity Agent at runtime and rotate automatically.

#### Tenant consumption pattern

**Block storage — default class:**

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: cache
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 10Gi
  # storageClassName omitted → gp3 (default)
```

**Block storage — high-IOPS class:**

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: gp3-iops
  resources:
    requests:
      storage: 100Gi
```

**S3 access via Pod Identity:**

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: log-analyzer
  namespace: team-c
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: log-analyzer
  namespace: team-c
spec:
  template:
    spec:
      serviceAccountName: log-analyzer # bound to the tenant's IAM role
      containers:
        - name: app
          image: ghcr.io/team-c/log-analyzer:1.0.0
          env:
            - name: BUCKET
              value: team-c-logs-dev
```

#### Platform ↔ tenant interactions

- **New bucket request.** Tenant opens a change request naming the bucket, access pattern (read-only / read-write), and target ServiceAccount. Platform applies a Terraform change adding the bucket + role + Pod Identity association. Tenant references the ServiceAccount on next deploy.
- **Bucket policy changes.** Owned by the platform. Tenants do not modify IAM policies inline.
- **PVC growth.** Tenants patch `spec.resources.requests.storage`; the CSI driver expands the underlying volume. No platform action required.
- **Data protection (target state).** Backup and snapshot policies are deferred; called out in Phase 00 as optional.

#### Demos

| Demo                | Storage used               | Illustrates                                                           |
| ------------------- | -------------------------- | --------------------------------------------------------------------- |
| `nginx-web`         | none                       | Stateless workload; no PVC, no bucket.                                |
| `to-do-app`         | `gp3-iops` PVC (Postgres)  | High-IOPS block storage on the `database` node class.                 |
| Team C log analyzer | S3 (read) via Pod Identity | Tenant-owned bucket, IAM role scoped to that bucket, no static creds. |

#### Open items

- **Bucket naming and tagging convention.** Needs to be codified before onboarding a second S3-using tenant, so buckets are consistently named, tagged with tenant / environment, and discoverable.
- **Templating the S3 wiring.** Bucket + IAM role + Pod Identity association is currently hand-written per tenant (see the Loki precedent). A Terraform module or ArgoCD `ApplicationSet` template would keep the onboarding process short.
- **Kyverno policy for PVC hygiene.** Enforce that every PVC declares an explicit `storageClassName` (avoids silent reliance on the default) and a size limit ceiling per namespace.
- **EFS.** Deferred until a shared-filesystem tenant lands.

### Phase 03 — Security capabilities

**Purpose.** Define the security contract between the platform and tenants. Two layers matter: **isolation** — what stops one tenant from affecting another — and **enforcement** — what stops a tenant from violating the platform's assumptions in the first place. Both are non-negotiable for a multi-tenant cluster.

#### Components in scope

| Concern           | Tooling                                           | Enforcement point                         |
| ----------------- | ------------------------------------------------- | ----------------------------------------- |
| Sensitive data    | External Secrets Operator + AWS Secrets Manager   | Runtime (`ExternalSecret` reconciliation) |
| Workload identity | EKS Pod Identity                                  | Runtime (SDK credential vending)          |
| TLS certificates  | cert-manager + Let's Encrypt `ClusterIssuer`      | Runtime (Certificate reconciliation)      |
| mTLS (east-west)  | Istio ambient (ztunnel)                           | Runtime (data plane)                      |
| Admission policy  | Kyverno `ClusterPolicy`                           | Admission (API server webhook)            |
| Network policy    | VPC CNI NetworkPolicy + Istio AuthorizationPolicy | Data plane                                |
| Resource fairness | `ResourceQuota` + `LimitRange`                    | Admission                                 |

#### Responsibility model

| Concern                                        | Platform | Tenant | Notes                                                                         |
| ---------------------------------------------- | :------: | :----: | ----------------------------------------------------------------------------- |
| Namespace creation and labels                  |    ✅    |        | Namespace carries a `team=<name>` label used by Kyverno and observability.    |
| Default-deny NetworkPolicy                     |    ✅    |        | Applied at namespace creation; tenants layer allow-rules on top.              |
| Additional NetworkPolicy / AuthorizationPolicy |          |   ✅   | Tenants own their internal ingress / egress rules.                            |
| `ResourceQuota` and `LimitRange`               |    ✅    |        | Baseline quota per namespace; tenants request increases.                      |
| `ClusterSecretStore` (AWS Secrets Manager)     |    ✅    |        | One `ClusterSecretStore`; tenants create `ExternalSecret` in their namespace. |
| Secret values in AWS Secrets Manager           |          |   ✅   | Tenants own their secret material; platform owns the vending path.            |
| Pod Identity role for tenant workloads         |    ✅    |        | Created per tenant on request (same pattern as Phase 02 S3 access).           |
| `ServiceAccount` → workload wiring             |          |   ✅   | Tenants set `spec.serviceAccountName`.                                        |
| Kyverno `ClusterPolicy` set                    |    ✅    |        | Cluster-wide; tenants cannot bypass or edit.                                  |
| Istio ambient enrollment (namespace label)     |    ✅    |        | Applied at namespace creation; enables mTLS without sidecars.                 |
| Pod-level `securityContext`                    |          |   ✅   | Tenants set non-root, drop capabilities, seccomp — Kyverno enforces baseline. |

#### Isolation model

Four layers, each enforced by a different mechanism. Any one alone is insufficient; together they form the tenant boundary.

| Layer        | Mechanism                                      | What it stops                                                                     |
| ------------ | ---------------------------------------------- | --------------------------------------------------------------------------------- |
| Namespace    | Kubernetes namespace + RBAC                    | Tenant A cannot `get / list / patch` Tenant B's objects.                          |
| Network (L3) | NetworkPolicy — default deny + selective allow | Tenant A pods cannot open TCP/UDP to Tenant B pods.                               |
| Network (L7) | Istio ambient AuthorizationPolicy + mTLS       | Even if L3 allows, workload identity is required; traffic is encrypted.           |
| Resources    | `ResourceQuota` + `LimitRange`                 | Tenant A cannot consume so much CPU / memory / PVC storage that Tenant B starves. |

Design principles:

- **Default deny, explicit allow.** Every NetworkPolicy is default-deny at namespace creation. Tenants add allow-rules for the ingress paths they need.
- **Identity is the primary boundary.** L7 policy is expressed in terms of ServiceAccount identity, not IP. IPs are ephemeral; identity is stable and encoded in the mTLS handshake.
- **Ambient mTLS is opt-in per namespace, mandatory for tenants.** Namespaces created by the onboarding flow are labeled `istio.io/dataplane-mode=ambient`; platform namespaces that predate ambient may not be enrolled.
- **Kyverno is the last line of defense.** If a Kyverno policy would fail closed and break the cluster, that is a signal the policy is too broad — but it is not a reason to loosen the tenant contract.

#### Admission-time contract (Kyverno)

The cluster ships a starter set of `ClusterPolicy` objects at [argocd/platform-capabilities/security/kyverno-policies/](argocd/platform-capabilities/security/kyverno-policies/). Every policy excludes platform namespaces (`kube-system`, `karpenter`, `istio-system`, `argocd`, etc.) so it applies to tenant namespaces only.

| Policy                              | Enforces                                                                             |
| ----------------------------------- | ------------------------------------------------------------------------------------ |
| `require-team-label`                | Every workload carries a `team` label — drives alerting, cost allocation, ownership. |
| `require-requests`                  | Every container declares CPU and memory `requests`.                                  |
| `require-probes`                    | Every container declares liveness and readiness probes.                              |
| `require-runbook-annotation`        | Every workload has a runbook URL annotation.                                         |
| `disallow-latest-tag`               | No `image: foo:latest`.                                                              |
| `disallow-privileged`               | No privileged containers, no host mounts of `/`.                                     |
| `disallow-host-namespace`           | No `hostNetwork`, `hostPID`, `hostIPC`.                                              |
| `restrict-image-registries`         | Images must come from the approved registry list.                                    |
| `httproute-hostname-scoped-to-team` | Tenants may only claim hostnames under their own subdomain.                          |

Follow-on policies scheduled below in _Open items_.

#### Secrets — consumption pattern

Tenants never handle raw AWS credentials or embed secret material in Git. The pattern is:

1. Platform ships a single `ClusterSecretStore` (`aws-secretsmanager`) pointing at AWS Secrets Manager in `ca-central-1`.
2. Tenant creates a secret in AWS Secrets Manager (via Terraform, console, or CLI) under a tenant-scoped prefix.
3. Tenant declares an `ExternalSecret` in their namespace referencing the secret name.
4. ESO reconciles the `ExternalSecret` into a native `Secret` in the tenant namespace, which the pod mounts as usual.

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-password
  namespace: team-b
spec:
  refreshInterval: 1h
  secretStoreRef:
    kind: ClusterSecretStore
    name: aws-secretsmanager
  target:
    name: db-password # native Secret created here
  data:
    - secretKey: password
      remoteRef:
        key: team-b/db/password
```

Access control at the AWS side is enforced by the `ClusterSecretStore`'s IAM role — its policy restricts `secretsmanager:GetSecretValue` to secrets under specific prefixes. A tenant cannot read another tenant's secrets even if they know the ARN.

#### mTLS — consumption pattern

Namespaces created by the onboarding flow are labeled `istio.io/dataplane-mode=ambient`. From that point:

- All pod-to-pod traffic within the mesh is transparently encrypted by ztunnel — no sidecar, no code change.
- Workload identity is expressed as a SPIFFE ID derived from the ServiceAccount.
- Tenants that want L7 policy (e.g., "only ServiceAccount X can call this Service") declare `AuthorizationPolicy` in their namespace.

#### Onboarding contract

When a tenant onboards, the platform performs the following in a single ArgoCD `ApplicationSet` reconciliation (target state — today parts are still manual):

1. Create namespace `<team>` with labels `team=<team>`, `istio.io/dataplane-mode=ambient`.
2. Apply default-deny `NetworkPolicy` (ingress and egress).
3. Apply baseline `ResourceQuota` and `LimitRange`.
4. Create the tenant's `ServiceAccount`(s) and any Pod Identity associations declared in the onboarding request.
5. Register the tenant's `Application` under `argocd/tenants/<team>/`.
6. Provision the tenant's Grafana folder and `PrometheusRule` label scope (Phase 05).

The tenant then owns everything inside that namespace: pods, services, PVCs, `HTTPRoute`s (scoped to their subdomain), `AuthorizationPolicy`, additional NetworkPolicy rules, and `ExternalSecret`s.

#### Platform ↔ tenant interactions

- **Quota increase.** Tenant opens a change request with justification. Platform patches the `ResourceQuota`; no restart required.
- **New AWS secret.** Tenant creates the secret in AWS Secrets Manager under their prefix and declares an `ExternalSecret`. No platform action if the prefix is already allow-listed.
- **New pod-to-pod policy.** Tenant declares NetworkPolicy / AuthorizationPolicy in their namespace. Platform does not review.
- **Cross-tenant call (rare).** Explicitly out of scope for the default contract. Requires platform review and a documented `AuthorizationPolicy` from both sides.

#### Open items

- **Onboarding automation.** Today namespace, quota, default-deny NP, and ArgoCD registration are partly manual. Target: a single `TeamNamespace` Helm chart driven by an `ApplicationSet` — one commit onboards a tenant.
- **Enforce `workload-class: platform` is tenant-forbidden.** Kyverno rule blocking tenant workloads from selecting the platform node class (called out in Phase 01).
- **Enforce explicit `storageClassName` on PVCs.** Called out in Phase 02.
- **Secret prefix contract.** Formalize the AWS Secrets Manager naming convention (`<team>/<component>/<key>`) and align the `ClusterSecretStore` role's IAM policy to those prefixes.
- **Namespace exclusion list is duplicated across every Kyverno policy** (9 places today). Extract to a shared ConfigMap / policy variable.
- **Baseline `PodSecurityContext` policy.** Add Kyverno rules to require `runAsNonRoot`, `readOnlyRootFilesystem` where feasible, and `capabilities.drop: [ALL]`.
- **Egress control.** Today NetworkPolicy handles ingress well; egress to the internet is unrestricted. Deferred; revisit when a tenant handles regulated data.

### Phase 04 — Network capabilities

**Purpose.** Define how external traffic reaches tenant workloads. Three pieces move together: **ingress** (Gateway API + Istio), **TLS** (cert-manager + Let's Encrypt), and **DNS** (external-dns + Route 53). The design goal is a single opinionated path — a tenant ships one `HTTPRoute` and gets a routable, TLS-terminated hostname without touching AWS, DNS, or certificates.

#### Components in scope

| Concern       | Tooling                                                      | Notes                                                                        |
| ------------- | ------------------------------------------------------------ | ---------------------------------------------------------------------------- |
| Ingress       | Gateway API + Istio ambient (`gatewayClassName: istio`)      | One shared `Gateway` on an internet-facing NLB.                              |
| Load balancer | AWS Load Balancer Controller (NLB, IP target type)           | NLB provisioned by the `Gateway` via infrastructure annotations.             |
| TLS           | cert-manager + Let's Encrypt (`letsencrypt-prod`) via DNS-01 | Wildcard cert for `*.arguswatcher.net`.                                      |
| DNS           | external-dns → Route 53                                      | Tenants declare hostnames on `HTTPRoute`; records are created automatically. |
| East-west     | Istio ambient (ztunnel)                                      | Covered in Phase 03.                                                         |

#### Responsibility model

| Concern                                             | Platform | Tenant | Notes                                                                              |
| --------------------------------------------------- | :------: | :----: | ---------------------------------------------------------------------------------- |
| `GatewayClass` and shared `Gateway`                 |    ✅    |        | One `Gateway` in `istio-ingress`, `gatewayClassName: istio`.                       |
| NLB lifecycle and annotations                       |    ✅    |        | Controlled by `Gateway.spec.infrastructure.annotations`.                           |
| DNS zone (`arguswatcher.net`)                       |    ✅    |        | Zone lives in Route 53; external-dns has write access.                             |
| Wildcard TLS certificate                            |    ✅    |        | `*.arguswatcher.net` issued by cert-manager, mounted on the shared `Gateway`.      |
| Team subdomain contract (`<team>.arguswatcher.net`) |    ✅    |        | Kyverno enforces that HTTPRoute hostnames match the tenant's subdomain.            |
| `HTTPRoute` in tenant namespace                     |          |   ✅   | Tenants declare their own routing rules, referencing the shared `Gateway`.         |
| Hostname choice (within their subdomain)            |          |   ✅   | Any hostname under `<team>.arguswatcher.net`.                                      |
| Path / header / method routing                      |          |   ✅   | `HTTPRoute` match rules are entirely tenant-owned.                                 |
| Custom domain (non-`arguswatcher.net`)              |    ⚠️    |   ⚠️   | Out of scope for the default contract; requires platform to add a listener + cert. |

#### Design principles

- **One shared Gateway, many `HTTPRoute`s.** Provisioning an NLB per tenant is expensive and slow. A single Gateway with `allowedRoutes.namespaces.from: All` lets any tenant namespace attach without ceremony.
- **TLS is a platform concern, not a tenant concern.** Tenants never handle certificates. The wildcard cert covers every tenant hostname under `arguswatcher.net`.
- **DNS is automatic, but scoped.** external-dns creates records for any `HTTPRoute` hostname, but Kyverno rejects hostnames outside the tenant's subdomain — so a tenant cannot silently hijack another tenant's DNS.
- **Gateway API, not `VirtualService`.** `HTTPRoute` is the tenant-facing surface. Mesh-specific resources (`VirtualService`, `DestinationRule`) remain available for platform use and for tenants who need advanced Istio features, but are not part of the default contract.
- **HTTP is redirected, not served.** The `Gateway` accepts port 80 for the ACME challenge and health checks; production traffic uses 443. Redirect from 80→443 is the tenant's `HTTPRoute` responsibility today; a shared redirect policy on the Gateway is an _Open item_.

#### Team subdomain contract

Each tenant gets a subdomain of the form `<team>.arguswatcher.net`. They may claim:

- `<team>.arguswatcher.net` — the apex of their subdomain, or
- `<anything>.<team>.arguswatcher.net` — any depth of sub-subdomain.

This is enforced at admission by the `httproute-hostname-scoped-to-team` Kyverno policy ([kyverno-policies/httproute-hostname-scoped-to-team.yaml](argocd/platform-capabilities/security/kyverno-policies/httproute-hostname-scoped-to-team.yaml)), which reads the namespace's `team` label and compares it against every hostname in the `HTTPRoute`.

#### Tenant consumption pattern

Tenants attach to the shared Gateway with an `HTTPRoute`. No `Service` annotations, no ingress annotations, no certificate manipulation.

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: web
  namespace: team-a
spec:
  parentRefs:
    - name: istio-ingress
      namespace: istio-ingress
  hostnames:
    - team-a.arguswatcher.net
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: web
          port: 80
```

On admission the platform automatically:

1. Kyverno checks that `team-a.arguswatcher.net` matches the namespace's `team` label.
2. external-dns creates an ALIAS record in Route 53 pointing to the shared NLB.
3. The shared TLS certificate already covers `*.arguswatcher.net`, so HTTPS works immediately with no per-tenant certificate action.

#### Platform ↔ tenant interactions

- **Add a hostname under an existing subdomain.** Tenant edits their `HTTPRoute`. No platform action.
- **New tenant subdomain.** Onboarding creates the `team` label and namespace; no DNS delegation change is needed because the whole zone is platform-managed and the wildcard cert already covers the subdomain.
- **Custom domain (`example.com`).** Out of the default contract. Platform must add a listener to the shared `Gateway`, a `Certificate` for the domain, and — if the tenant does not delegate the zone — coordinate DNS with the domain owner.
- **Traffic spike / NLB scaling.** NLBs scale transparently; no tenant or platform action.
- **Gateway maintenance / restart.** Ambient traffic drains via ztunnel; short reconnect windows are the only tenant-visible symptom.

#### Demos

| Demo            | Hostname                                                             | Illustrates                                                       |
| --------------- | -------------------------------------------------------------------- | ----------------------------------------------------------------- |
| `nginx-web`     | `team-a.arguswatcher.net`                                            | Minimal `HTTPRoute` on the default team subdomain.                |
| `to-do-app`     | `todo.team-b.arguswatcher.net` (API), `team-b.arguswatcher.net` (UI) | Two hostnames under one subdomain, same `HTTPRoute` or two.       |
| Team C analyzer | `analyzer.team-c.arguswatcher.net`                                   | Ingress not required for the core function; used for a status UI. |

#### Open items

- **Shared HTTP→HTTPS redirect.** Today the redirect is per-tenant `HTTPRoute` boilerplate. A single `HTTPRoute` in `istio-ingress` matching every hostname on port 80 with a `RequestRedirect` filter would remove that boilerplate.
- **Rate limiting.** No cluster-level or per-tenant rate limits at the edge. Deferred until abuse is observed.
- **WAF.** No AWS WAF association on the NLB (NLB does not support WAF directly; would require an ALB in front or CloudFront). Out of scope for the default contract.
- **Internal-only routes.** Every hostname today is internet-facing. A second `Gateway` on an internal NLB (with `aws-load-balancer-scheme: internal` and a private ClusterIssuer / self-signed CA) would let tenants expose services to the VPC only.
- **Custom-domain onboarding.** Formalize the process for a tenant to bring their own domain: DNS delegation vs. cert-manager DNS-01 with the tenant's provider vs. platform-owned zone.
- **`allowedRoutes.namespaces.from: All` is permissive.** Combined with the Kyverno subdomain check it is safe, but the belt-and-braces version is a namespace `Selector` matching only namespaces with a `team` label.

### Phase 05 — Observability

**Purpose.** Extend the existing single-tenant observability stack (Prometheus + Grafana + Loki + Alloy + Alertmanager) into a multi-tenant model — one shared stack, per-tenant scoping. The platform keeps a single pane of glass; tenants see only their own signals.

The design deliberately avoids running a Prometheus / Loki / Grafana per tenant. That gets expensive quickly and duplicates operational load without materially better isolation for the current tenant scale.

#### Components in scope

| Signal     | Tooling                              | Storage                                               | Notes                                                                                        |
| ---------- | ------------------------------------ | ----------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| Metrics    | Prometheus (kube-prometheus-stack)   | `gp3` PVC, 7 d / 15 GiB retention                     | Discovers `ServiceMonitor`, `PodMonitor`, `PrometheusRule` cluster-wide.                     |
| Logs       | Loki + Grafana Alloy DaemonSet       | S3 (see [infra/16-eks-loki.tf](infra/16-eks-loki.tf)) | Alloy tails pod logs and labels them with `namespace`, `pod`, `container`, `team`, `app`.    |
| Dashboards | Grafana                              | `gp3` PVC                                             | Sidecar discovers dashboards from `ConfigMap`s cluster-wide via `grafana_dashboard=1` label. |
| Alerts     | Alertmanager → Slack                 | `gp3` PVC                                             | Discovers `AlertmanagerConfig` by `alertmanagerConfig=platform` label.                       |
| Traces     | _(deferred — see Phase 00 optional)_ | —                                                     | Ambient mesh emits trace-ready metrics; no collector today.                                  |

#### Responsibility model

| Concern                                              | Platform | Tenant | Notes                                                                                |
| ---------------------------------------------------- | :------: | :----: | ------------------------------------------------------------------------------------ |
| Prometheus / Loki / Grafana / Alertmanager lifecycle |    ✅    |        | Single shared stack in `monitoring`.                                                 |
| Alloy DaemonSet log collection                       |    ✅    |        | Automatic — no tenant action to get logs shipped.                                    |
| Platform-level dashboards and alerts                 |    ✅    |        | Cluster health, node pressure, ArgoCD sync, Istio, etc.                              |
| `ServiceMonitor` / `PodMonitor` for tenant apps      |          |   ✅   | Tenants declare scrape targets in their namespace.                                   |
| `PrometheusRule` for tenant SLOs / alerts            |          |   ✅   | Must carry a `team=<team>` label — drives Alertmanager routing.                      |
| Grafana dashboards for tenant apps                   |          |   ✅   | ConfigMap with `grafana_dashboard=1` label + `grafana_folder` annotation.            |
| `AlertmanagerConfig` for team routing                |    ✅    |        | Platform ships one per team as part of onboarding; tenants request receiver changes. |
| SLOs / runbooks / on-call for tenant apps            |          |   ✅   | Tenants own signals for their code.                                                  |

#### Isolation model

Four axes of isolation. All achieved by convention + label discovery, not by running separate stacks.

| Axis       | Mechanism                                                                                                                                                          | What a tenant can and cannot see                                                     |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------ |
| Metrics    | `team=<team>` label injected on `ServiceMonitor` / `PodMonitor` / `PrometheusRule`; Grafana data source uses PromQL label matchers.                                | Tenants query only `{team="team-a"}`; enforced at the dashboard / data source level. |
| Logs       | Alloy relabels every log line with `team` from the pod label ([alloy.yaml](argocd/platform-capabilities/observability/alloy.yaml)); Loki queries filter by `team`. | Same model — LogQL queries carry `{team="team-a"}`.                                  |
| Dashboards | Grafana folders per team; each team's ConfigMaps carry `grafana_folder: <team>`.                                                                                   | Tenants edit only their own folder; Grafana RBAC enforces this once SSO is in place. |
| Alerts     | Alertmanager routes on the `team` label; per-team receivers land in `#<team>-alerts` / `#<team>-oncall`.                                                           | Tenants receive only their own alerts; platform receives everything critical.        |

Design principles:

- **Isolation by label, not by namespace or instance.** Every signal — metric, log line, alert — carries the `team` label. Every query, dashboard, and alert route pivots on it.
- **Cluster-wide discovery, label-based selection.** The stack is configured with `serviceMonitorSelectorNilUsesHelmValues: false` and equivalent for `PodMonitor`, `Probe`, `PrometheusRule`. Tenants add resources in their own namespace; discovery is automatic.
- **Grafana's sidecar discovery is the tenant plumbing for dashboards.** A tenant ships a `ConfigMap` with `grafana_dashboard=1` and `grafana_folder: <team>`; the dashboard appears in their folder without a platform ticket.
- **Alertmanager routing is where multi-tenancy is enforced most visibly.** The `matchers: [team="<team>"]` route is created at onboarding, along with the tenant's Slack channels; the tenant does not edit the top-level Alertmanager config.

#### Tenant consumption pattern

**Metrics — scrape a workload:**

```yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: web
  namespace: team-a
  labels:
    team: team-a
spec:
  selector:
    matchLabels:
      app: web
  endpoints:
    - port: metrics
      interval: 30s
```

**Metrics — alert on latency:**

```yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: web-slo
  namespace: team-a
  labels:
    team: team-a
spec:
  groups:
    - name: web
      rules:
        - alert: WebLatencyHigh
          expr: histogram_quantile(0.99, sum by (le) (rate(http_request_duration_seconds_bucket{team="team-a"}[5m]))) > 0.5
          for: 10m
          labels:
            team: team-a
            severity: warning
          annotations:
            runbook_url: https://runbooks/team-a/web-latency
```

**Dashboards — ship as a `ConfigMap`:**

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: web-dashboard
  namespace: team-a
  labels:
    grafana_dashboard: "1"
  annotations:
    grafana_folder: team-a
data:
  web.json: |
    { ... dashboard JSON, PromQL queries scoped to team="team-a" ... }
```

**Logs — no tenant action.** Alloy already tails every pod's stdout and applies the `team` label from the pod's `team` label. In Grafana, a tenant queries `{team="team-a"} |= "error"`.

#### Alerting flow

```
tenant PrometheusRule (labels: team=team-a, severity=critical)
  → Prometheus fires
  → Alertmanager receives, groups by (alertname, team, severity)
  → route matcher team=team-a → receiver team-a-oncall
  → Slack #team-a-oncall
```

Critical severity from any team also fans out to `#platform-oncall` (the top-level `continue: true` route in [prometheus-stack.yaml](argocd/platform-capabilities/observability/prometheus-stack.yaml)), so the platform on-call has cluster-wide visibility.

#### Platform ↔ tenant interactions

- **New tenant onboards.** Platform creates the tenant's `AlertmanagerConfig` (or top-level route) with matchers on `team=<team>` and receivers pointing to the tenant's Slack channels. Tenant gets a Grafana folder and default data source access.
- **Tenant adds a dashboard.** No platform action — ConfigMap with the right label is discovered automatically.
- **Tenant adds an alert.** No platform action — `PrometheusRule` with the right `team` label routes to the tenant's Slack.
- **Tenant needs more Prometheus retention or higher cardinality.** Requires a platform change (retention is cluster-wide). Deferred to the roadmap; long-term storage (Thanos/Mimir) is the real fix.
- **Grafana access.** Today Grafana runs with an admin login; per-team RBAC lands with SSO (Phase 06 or later — noted in the roadmap).

#### Demos

| Demo            | Signals shown                                                    | Illustrates                                                                |
| --------------- | ---------------------------------------------------------------- | -------------------------------------------------------------------------- |
| `nginx-web`     | Basic HTTP metrics + logs                                        | Zero-config log ingestion; a minimal `ServiceMonitor`.                     |
| `to-do-app`     | App metrics + Postgres metrics + latency alert                   | Multi-signal dashboard, per-team folder, alert routed to `#team-b-alerts`. |
| Team C analyzer | Log queries against tenant-owned S3 dataset + processing metrics | LogQL scoped by `team=team-c`; alert on job failures.                      |

#### Open items

- **Grafana per-team RBAC.** Requires SSO (GitHub OAuth or IAM Identity Center). Today Grafana admin sees everything and there is no user-level scoping — tenant isolation of dashboards is by convention only.
- **PromQL / LogQL query enforcement.** The label-based isolation model relies on tenants querying with the right `team` matcher. A future Prometheus / Loki multi-tenancy layer (Cortex, Mimir, Loki multi-tenancy header) would enforce this in the data path.
- **`AlertmanagerConfig` per team.** Today the top-level Alertmanager config carries every team's routing. Migrate to per-namespace `AlertmanagerConfig` resources (discovery is already wired: `alertmanagerConfig=platform` selector). One resource per tenant, shipped by the onboarding primitive.
- **Long-term metrics storage.** 7-day / 15 GiB retention is enough for the current scope; not enough for capacity planning or postmortems beyond a week. Thanos / Mimir when scope warrants.
- **Distributed tracing.** Ambient mesh emits trace-ready metrics; no Tempo / OTel collector deployed. Add when a tenant asks.
- **Cost visibility.** OpenCost with `team` labels would close the loop between the tenancy contract and the AWS bill.
- **Dashboard-as-code discipline.** A team can currently ship arbitrary JSON. A minimal linter (required title, required data source, required team folder annotation) would keep the shared Grafana coherent.

### Phase 06 — Delivery

**Purpose.** Define the delivery contract: how manifests get from Git into the cluster, and how tenants roll out changes safely. The platform commits to a single opinionated GitOps path (**ArgoCD**) and offers progressive delivery (**Argo Rollouts**) as an opt-in capability tenants may or may not consume.

#### Components in scope

| Concern              | Tooling                                       | Notes                                                                                                                                                          |
| -------------------- | --------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Continuous delivery  | ArgoCD (self-managing, root-of-roots pattern) | Single instance; every platform and tenant resource reconciles through it.                                                                                     |
| Progressive delivery | Argo Rollouts                                 | Optional. Tenants opt in by using `Rollout` instead of `Deployment`.                                                                                           |
| Canary analysis      | Argo Rollouts `ClusterAnalysisTemplate`       | Shared templates (success rate, p95 latency, error-budget burn) at [argo-rollouts-resources/](argocd/platform-capabilities/delivery/argo-rollouts-resources/). |
| Tenant boundaries    | ArgoCD `AppProject`                           | One `AppProject` per tenant; scopes source repos, destination namespaces, and allowed resource kinds.                                                          |

#### Responsibility model

| Concern                                   |             Platform              | Tenant | Notes                                                                                                                  |
| ----------------------------------------- | :-------------------------------: | :----: | ---------------------------------------------------------------------------------------------------------------------- |
| ArgoCD lifecycle                          |                ✅                 |        | Self-managed; upgraded via GitOps like every other capability.                                                         |
| Root ArgoCD `Application` (`00-root`)     |                ✅                 |        | Points at `argocd/bootstrap/` and fans out to platform + tenants.                                                      |
| Platform `Application`s                   |                ✅                 |        | Under `argocd/platform-capabilities/`; sync-wave-ordered.                                                              |
| Per-tenant `AppProject`                   |                ✅                 |        | Created at onboarding; scopes source repos and destinations.                                                           |
| Tenant `Application`(s)                   | ✅ (registration) / ✅ (contents) |   ✅   | Platform registers the top-level `Application` under `argocd/tenants/<team>/`; tenant owns the manifests it points at. |
| Argo Rollouts controller                  |                ✅                 |        | Cluster-wide; watches every `Rollout` regardless of namespace.                                                         |
| `ClusterAnalysisTemplate`s                |                ✅                 |        | Shared success rate / latency / error-budget-burn templates.                                                           |
| `Rollout` spec (steps, analysis, weights) |                                   |   ✅   | Tenants own the delivery strategy for their own workloads.                                                             |
| Container image build and push            |                                   |   ✅   | Out of scope for the platform. Tenants use whatever CI produces images.                                                |

#### Design principles

- **GitOps is mandatory, not optional.** Every resource in the cluster reconciles through ArgoCD. `kubectl apply` is a debugging tool, not a delivery mechanism.
- **Root-of-roots pattern.** A single `00-root` `Application` bootstraps the cluster; it is the only object applied by hand (or by Terraform). Everything else — platform capabilities, tenant apps — is discovered from Git.
- **`AppProject` is the delivery-layer tenant boundary.** It restricts which repos a tenant can source from and which namespaces / cluster-scoped kinds their `Application`s can touch. This is the last line of defense before Kyverno.
- **Progressive delivery is a shared capability, not a mandate.** The platform runs one Argo Rollouts controller and ships reusable `ClusterAnalysisTemplate`s. Tenants who want blue/green or canary use `Rollout`; tenants who don't stay on `Deployment` — no cost either way.
- **Image build is out of scope.** The platform delivers manifests, not code. CI (GitHub Actions, whatever) produces images; the platform reconciles the manifests that reference them.

#### Tenant consumption pattern

**Registering a tenant app** — platform creates one `Application` per tenant under `argocd/tenants/<team>/`:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: team-a-web
  namespace: argocd
spec:
  project: team-a # tenant's AppProject
  source:
    repoURL: https://github.com/team-a/web
    targetRevision: main
    path: deploy/
  destination:
    server: https://kubernetes.default.svc
    namespace: team-a
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

**Basic Deployment** — tenant ships plain manifests, no progressive delivery:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: team-a
  labels:
    team: team-a
spec:
  replicas: 3
  # ... standard Deployment ...
```

**Progressive delivery** — tenant opts into canary with shared analysis:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: web
  namespace: team-a
  labels:
    team: team-a
spec:
  strategy:
    canary:
      steps:
        - setWeight: 20
        - analysis:
            templates:
              - templateName: success-rate
                clusterScope: true
            args:
              - name: service
                value: web-canary
              - name: namespace
                value: team-a
        - setWeight: 50
        - pause: { duration: 5m }
        - setWeight: 100
  # ... standard Rollout spec ...
```

The tenant references shared `ClusterAnalysisTemplate`s (`success-rate`, `latency-p95`, `error-budget-burn`) — they do not write their own PromQL. If a tenant needs a custom analysis, they may ship a namespace-scoped `AnalysisTemplate`.

#### Platform ↔ tenant interactions

- **Tenant onboards.** Platform creates the `AppProject`, one initial `Application`, the tenant namespace, and its Grafana folder. Tenant pushes to their repo; ArgoCD reconciles.
- **Tenant adds a second app.** Tenant opens a PR against `argocd/tenants/<team>/` adding another `Application`. Platform merges after checking it stays inside the tenant's `AppProject` scope.
- **Tenant wants canary.** Tenant swaps `Deployment` for `Rollout`, references the shared analysis templates. No platform change.
- **Tenant wants a custom analysis metric.** Tenant ships a namespace-scoped `AnalysisTemplate`. No platform change unless it requires a data source the tenant does not have access to.
- **ArgoCD outage.** Existing workloads keep running; new syncs are queued. Recovery is a platform on-call action.

#### Boundary at the `AppProject`

The `AppProject` is deliberately narrow. For each tenant it declares:

- `sourceRepos` — only the tenant's own repositories (and the platform repo, if they consume shared charts).
- `destinations` — only their own namespace.
- `clusterResourceWhitelist` — nothing, except `Namespace` if the tenant is allowed to create their own subordinate namespaces. Kyverno enforces the same.
- `namespaceResourceWhitelist` — `*` inside their namespace.

This means a tenant `Application` cannot accidentally target `kube-system`, cannot install a `ClusterRole`, and cannot pull from a repo the platform has not approved.

#### Demos

| Demo            | Delivery style                                         | Illustrates                                                                 |
| --------------- | ------------------------------------------------------ | --------------------------------------------------------------------------- |
| `nginx-web`     | Plain `Deployment`, ArgoCD auto-sync                   | Simplest GitOps path; no Rollouts.                                          |
| `to-do-app`     | `Rollout` with canary + shared `success-rate` analysis | Progressive delivery on the API tier; Postgres stays a `StatefulSet`.       |
| Team C analyzer | `CronJob`, ArgoCD auto-sync                            | Non-serving workload — Rollouts irrelevant; GitOps still owns the schedule. |

#### Open items

- **Onboarding automation.** Today creating a tenant means hand-writing `AppProject` + initial `Application` + namespace + observability wiring. Target: one `TeamNamespace` `ApplicationSet` fanning out from a single tenant declaration (see Phase 03 open item).
- **ArgoCD SSO.** Admin login only today; per-team RBAC in ArgoCD (view-only on their `Application`s, no edit) requires SSO. Same dependency as Grafana SSO in Phase 05.
- **Sync notifications.** ArgoCD Notifications integration → Slack (`#<team>-alerts` on `OutOfSync` / `Degraded`), sharing the routing model from Phase 05.
- **`ApplicationSet` templates for tenant apps.** A tenant declaring `{ team: team-a, repos: [web, api] }` in one place would beat maintaining N `Application`s by hand.
- **Image update automation.** Argo CD Image Updater or Renovate on tenant repos — currently out of scope; tenants bump image tags themselves.
- **Rollback drills.** No documented rollback playbook. `argocd app rollback` works; needs to be part of the onboarding manual.

### Phase 07 — Onboarding

**Purpose.** Consolidate every decision from Phases 01–06 into a single tenant-facing manual. A new team should be able to read this section, hand the platform a short onboarding request, and ship a working service the same day.

#### The onboarding contract at a glance

**What the tenant brings:**

1. A **team name** (`<team>`), used as the namespace name, subdomain, alert-routing key, log/metric label, and Grafana folder.
2. A **Git repository** containing their manifests.
3. Optional: **AWS resources** they want access to (S3 buckets, Secrets Manager entries) and the ServiceAccount name they will use.
4. Optional: **Slack channels** for alert routing (`#<team>-alerts`, `#<team>-oncall`).

**What the platform provisions** — in a single GitOps commit against `argocd/tenants/<team>/`:

| #   | Resource                                                                                        | Reference phase |
| --- | ----------------------------------------------------------------------------------------------- | --------------- |
| 1   | Namespace with `team=<team>` and `istio.io/dataplane-mode=ambient` labels                       | Phase 03        |
| 2   | Default-deny `NetworkPolicy` (ingress + egress)                                                 | Phase 03        |
| 3   | Baseline `ResourceQuota` and `LimitRange`                                                       | Phase 03        |
| 4   | ArgoCD `AppProject` scoped to the tenant's repos and namespace                                  | Phase 06        |
| 5   | ArgoCD `Application`(s) pointing at the tenant's repo                                           | Phase 06        |
| 6   | Grafana folder (`grafana_folder: <team>`) and Alertmanager route to the tenant's Slack channels | Phase 05        |
| 7   | (Optional) IAM role + Pod Identity association for AWS access                                   | Phases 02, 03   |

The tenant's day-one work is a single PR against their own repo containing manifests scoped to their namespace.

#### What tenants consume — the one-page cheat sheet

| Need                        | How the tenant asks for it                                                                                                         |
| --------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| Default node placement      | Nothing — `general` is the default.                                                                                                |
| Database node placement     | `nodeSelector: { workload-class: database }` + toleration on `workload-class=database:NoSchedule`.                                 |
| GPU node placement          | `nodeSelector: { workload-class: gpu }` + toleration + `resources.limits.nvidia.com/gpu: N`.                                       |
| Default block storage       | `PVC` with no `storageClassName` → `gp3`.                                                                                          |
| High-IOPS block storage     | `PVC` with `storageClassName: gp3-iops`.                                                                                           |
| S3 access                   | Onboarding request lists bucket + ServiceAccount name; tenant sets `spec.serviceAccountName` on the pod.                           |
| Secret from Secrets Manager | `ExternalSecret` referencing `ClusterSecretStore/aws-secretsmanager` and a key under `<team>/…`.                                   |
| Public hostname             | `HTTPRoute` with `parentRefs: istio-ingress/istio-ingress` and any hostname under `<team>.arguswatcher.net`.                       |
| TLS certificate             | Automatic — wildcard cert covers every `*.arguswatcher.net` hostname.                                                              |
| DNS record                  | Automatic — external-dns creates the record on `HTTPRoute` admission.                                                              |
| Scrape a workload           | `ServiceMonitor` with `team=<team>` label in the tenant namespace.                                                                 |
| Alert                       | `PrometheusRule` with `team=<team>` label; routes to the tenant's Slack automatically.                                             |
| Dashboard                   | `ConfigMap` with label `grafana_dashboard: "1"` and annotation `grafana_folder: <team>`.                                           |
| Logs                        | Nothing — Alloy tails every pod's stdout and labels lines with `team`. Query with `{team="<team>"}` in Loki.                       |
| Continuous deploy           | Automatic — the tenant's ArgoCD `Application` reconciles on every commit.                                                          |
| Progressive delivery        | Swap `Deployment` for `Rollout`; reference shared `ClusterAnalysisTemplate`s (`success-rate`, `latency-p95`, `error-budget-burn`). |

Anything not on this list is either owned by the platform (tenants do not touch it) or explicitly out of scope (see Phase 00 optional / roadmap items).

#### Guardrails a tenant will hit

These are the admission-time rules that reject a manifest. Knowing them up front saves a support round-trip.

- Every workload must carry a `team` label (Phase 03).
- Every container must declare CPU/memory `requests` and liveness/readiness probes (Phase 03).
- Every workload must carry a runbook URL annotation (Phase 03).
- No `image: *:latest`, no privileged containers, no host namespaces, images must come from approved registries (Phase 03).
- `HTTPRoute` hostnames must fall under `<team>.arguswatcher.net` (Phase 04).
- Tenants may not select `workload-class: platform` — _(target state, once the Kyverno rule lands)_ (Phase 01).

#### Multi-team demo — three teams onboard in parallel

Three tenants, each exercising a different slice of the platform. All onboard through the same primitive; the differences show up in what each tenant declares.

##### Team A — stateless web app

**Profile:** simplest possible workload. One `Deployment`, one `Service`, one `HTTPRoute`. Exercises the default path and nothing else.

**Onboarding request:**

- team = `team-a`
- repo = `github.com/team-a/web`
- no S3, no secrets, no custom Slack (defaults to `#team-a-alerts`)

**Tenant manifests (excerpt):**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: team-a
  labels: { team: team-a, app: web }
  annotations: { runbook_url: "https://runbooks/team-a/web" }

  replicas: 3
  template:
    spec:
      containers:
        - name: web
          image: ghcr.io/team-a/web:1.4.0
          ports: [{ containerPort: 8080, name: http }]
          resources: { requests: { cpu: 100m, memory: 128Mi } }
```

**Capabilities exercised:** compute (`general`), ingress, TLS, DNS, logs, metrics scrape (once a `ServiceMonitor` is added), GitOps.

##### Team C — log analysis reading from S3

**Profile:** batch workload. `CronJob` that reads objects from a tenant-owned S3 bucket, processes them, writes results back. Exercises Pod Identity, S3, and observability without needing ingress.

**Onboarding request:**

**Platform provisions (Terraform):**

#### Roadmap — what tenants should know is coming

Called out in earlier phases; grouped here so tenants see the trajectory, not gaps.

- **Onboarding automation via `TeamNamespace` `ApplicationSet`** — collapses the platform PR to a single tenant declaration.
- **SSO for Grafana and ArgoCD** — per-team RBAC, closes the "isolation by convention" gap for dashboards and app views.
- **Query-path multi-tenancy for metrics and logs** (Mimir / Loki multi-tenancy) — enforces the `team` label at the data path, not just at the resource level.
- **Custom-domain and internal-only ingress** — for tenants that need `example.com` or VPC-only services.
- **Egress control** — for tenants that handle regulated data.
- **Backup / DR (Velero), long-term metrics (Thanos/Mimir), tracing (Tempo), cost visibility (OpenCost)** — capability roadmap.
- **Image signing / verification** — supply-chain hardening when a signing pipeline exists.

#### Non-goals — worth naming so scope reads as intentional

- **Data platform as a service** (managed Postgres, Redis, Kafka). Tenants run their own on `database`-class nodes or use AWS-managed services.
- **Multi-cluster / multi-region.** One cluster; a separate project owns the cluster-per-environment story.
- **Developer portal (Backstage).** The onboarding manual lives in this document, not a UI.
- **Chaos engineering.** High effort, tangential to the core tenancy story.
